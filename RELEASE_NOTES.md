## v1.3.1 — 2026-03-31

### 🐛 修复

#### 第四轮：排序键字段名错误（`final_score` vs `total_score`）
- **问题**：每场预测结果都固定为 `[1, 2, 3]`，无论哪场都一样
- **根因**：`daily_scheduler.py` 中排序键使用 `h.get("final_score", 0)`，
  但 `analyze_race.py` 实际输出的字段名是 `total_score`。
  所有马的 `final_score` 都返回默认值 0 → 得分相同 →
  Python 稳定排序保持原始列表顺序 → 马号 1, 2, 3 永远排在前面
- **修复**：`daily_scheduler.py` 第 266、273 行：`final_score` → `total_score`
- **验证**：修复后各场马号完全多样化，评分差异明显（37–62 分）

#### 第三轮：后备马混入 + 马名提取失败
- **问题**：预测结果混杂后备马（如傳承效應、飛雲），且马号与马名无法对应
- **根因**：
  1. TR 正则 `class="[^"]*f_fs\d+[^"]*"` 同时匹配正选马（`f_tac f_fs13`）和后备马（`bg_white f_tac f_fs13`）
  2. 马名提取正则 `href="...horse\?horseid=[^"]+[^>]*>` 中，`[^"]+` 在 href 闭合 `"` 后遇到紧跟的 `onclick=`（中间无空格），贪心匹配导致失败
- **修复**：
  1. TR 匹配改为精确 `class="f_tac f_fs13"`（排除 bg_white 后备马）
  2. 所有链接提取改用 `[^>]*>` 绕过 onclick 干扰：`r'horseid=([A-Za-z0-9_]+)[^>]*>([^<]+)</a>'`
- **验证**：各场正选马完全独立（无后备），马号与马名一一对应

## v1.3.0 — 2026-03-30

### 🆕 新增
- **每日自动化调度系统**（`daily_scheduler.py`）：
  - 每天 14:30 自动检测明天是否为赛马日（HKJC 排位表检测）
  - 若是赛马日，批量预测所有场次并保存预测存档到 `.archive/`
  - 每天 09:00 自动抓取昨日实际赛果，运行回测，输出精度指标
  - 独赢命中率 / 前3命中率 / 场次级别逐场对比
- **自我进化分析引擎**（`daily_scheduler.py` 内置）：
  - 根据回测结果自动识别系统性偏差（高估/低估模式）
  - 生成结构化进化建议（`type` / `priority` / `dimension` / `code_change`）
  - 进化建议仅输出供用户审阅，**不自动修改 Skill**
  - 报告保存到 `.evolution/evolution_YYYY-MM-DD_VENUE.md`
- **进化建议应用工具**（`apply_evolution.py`）：
  - 用户确认某条建议后，安全应用到 `analyze_race.py`（含自动备份）
  - 支持回滚：`--rollback` 还原最近一次改动
  - 应用历史记录：`--history` 查看所有已应用改动
- **预测存档机制**：所有预测结果保存为 JSON，供日后横向对比分析

### 🔧 变更
- `dump_race.py` 硬编码绝对路径 → 改为相对路径（安全审计修复）

### 🏗️ 目录结构新增
```
.archive/     预测存档 + 回测报告 JSON
.evolution/   进化建议报告 MD + 应用历史
.backups/     每次 apply_evolution 的 analyze_race.py 备份
```

## v1.2.0 — 2026-03-30

### 🆕 新增
- **骑师动态评分**（`score_jockey`）：不再使用固定值50，改为从该马历史战绩中动态统计：
  - 优先路径：统计「该骑师骑乘本马」的胜率/前3率（≥2场样本）
  - 退化路径：与全场骑师平均前3率对比（1场样本）
  - 兜底：首次骑乘本马返回中性偏低值46；无历史数据返回50
- **练马师动态评分**（`score_trainer`）：逻辑同上，需≥3场样本；换新练马师给42（高不确定性）
- **历史战绩新增骑师/练马师字段**：`parse_horse_history()` 现在从每条历史记录中提取 `jockey`（列10）和 `trainer`（列9）字段，写入 history 结构供评分函数使用

### 🔧 变更
- **权重调整**（释放6%给历史维度，降低骑师/练马师权重）：

| 维度 | 旧权重 | 新权重 | 原因 |
|------|--------|--------|------|
| `history_same_condition` | 15% | **18%** | ↑ 强化数据导向 |
| `history_same_venue` | 10% | **13%** | ↑ 强化数据导向 |
| `jockey` | 8% | **5%** | ↓ 次要因素降权 |
| `trainer` | 7% | **4%** | ↓ 次要因素降权 |

### ✅ 验证
- 实测场次：2026/03/29 沙田第1场（14匹，全缓存秒级完成）
- 骑师分样例：1号部族高手骑师分70（周俊樂1场赔率缩水），11号劍無情骑师分75（艾兆禮高前3率）
- 练马师分样例：10号飛躍凱旋练马师分70（桂福特带本马6场，高前3率）

## v1.1.0 — 2026-03-30

### 🆕 新增
- **磁盘缓存系统**：所有 HKJC 数据请求自动缓存到 `.cache/`，按 URL 哈希命名
  - 赛果缓存7天、马匹历史24小时、赔率5分钟，有效避免重复网络请求
  - 新增 `--force-refresh` 参数：强制跳过缓存重新抓取
  - 新增 `--clear-cache` 参数：清除当前场次缓存后重新分析
  - 新增 `--cache-stats` 参数：查看缓存目录统计（文件数/大小/最旧条目）
- **Softmax 概率归一化**：弃用简单线性归一化，改用 Softmax（温度参数 1.5）+单马上限约束（50%），避免极端偏差
- **场景自适应权重**：新增 `--scenario` 参数，支持 `normal/newcomer/class_down/class_up` 四种场景
- **数据充足度标注**：每匹马输出 ⭐/⭐⭐/⭐⭐⭐ 充足度，提示评分可信程度
- **冷门关注逻辑**：同时满足赔率>15、有同条件前3记录、赔率未拉长、班次适配度≥50，自动标记 ⚡ 冷门关注
- **场地/距离/赛道权重调整**：跑马地+3%档位、1800m+长途+5%配速、泥地赛道专项权重
- **中文参数支持**：`--venue` 接受 `沙田/跑马地` 中文输入，自动映射为 `ST/HV`
- **历史战绩精准匹配**：`parse_horse_history()` 实现完整解析，支持途程、场地、班次、走位等字段；同条件容差±200m

### 🔧 变更
- 历史战绩权重拆分：从单一"历史"维度拆为「同距离同场地」+「同场地」+「班次适配度」三个独立子维度
- 赔率权重拆分：从单一"赔率"维度拆为「临场赔率绝对值」+「赔率走势变化」两个子维度
- 跑法推导：从历史走位自动推导马匹跑法（领跑/后追/均速），用于配速×场地状况评分匹配

### ✅ 验证
- 实测场次：2026/03/29 沙田第1场，14匹马历史全部成功抓取
- 同条件评分从默认40提升为实际计算值，数据充足度准确反映历史场次数量
- 冷门关注准确识别3匹马（13号/3号/14号）
