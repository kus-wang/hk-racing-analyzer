## v1.4.4 — 2026-04-02

### 💰 投注建议模块重构（AI 自主判断）

**背景**：原有投注建议仅为固定格式输出（独赢/连赢/三重彩），缺少判断逻辑和各投注类型的完整说明。

**变更内容**：

- `SKILL.md` 新增完整投注推荐逻辑（Step 4B/4C/4D）：
  - 独赢（Win）：Top1 概率 >= 30% 或赔率 <= 4 倍时推荐
  - 位置（Place）：Top3 任一匹概率 >= 18% 时推荐
  - 连赢（Quinella）：Top1+Top2 概率和 >= 50% 且差距 <= 20% 时推荐
  - 位置连赢（PQ）：Top3 概率分散（差值 > 25%）时推荐三组全包
  - 三重彩（Trio）：Top3 均 >= 10% 且和 >= 45% 时推荐
  - **最推荐方式**：根据概率分布特征自动判断最佳投注类型

- `SKILL.md` 同步修正：
  - 回测自动化时间 `09:00 → 23:30`（中英文版）
  - 专家预测权重 `5-10% → 4%`

### 🏗️ 重构：模块化拆分

**背景**：`analyze_race.py` 代码量达到 1778 行（54KB），功能耦合严重，不利于后期维护和功能扩展。

**重构方案**：将单文件拆分为 9 个模块，各司其职：

| 模块 | 文件 | 职责 |
|:-----|:-----|:-----|
| 配置 | `config.py` | URL常量、缓存TTL、权重默认值、场地/状况映射 |
| 权重 | `weights.py` | 场景/场地/距离适配的动态权重计算 |
| 概率 | `probability.py` | Softmax归一化概率计算 |
| 评分 | `scoring.py` | 所有评分函数（历史/赔率/配速/骑师/贴士等） |
| 缓存 | `cache.py` | 磁盘缓存读写、TTL过期、统计清理 |
| 抓取 | `fetch.py` | HTTP请求、Playwright动态加载、马匹历史/贴士指数 |
| 解析 | `parse.py` | 排位表、马匹历史战绩HTML解析 |
| 分析 | `analyze.py` | 单匹马多维度综合评分 |
| 输出 | `output.py` | Markdown报告格式化 |
| 入口 | `main.py` | CLI解析、主流程编排 |

**向后兼容**：`analyze_race.py` 保留为入口兼容层，直接调用 `main.py`，CLI用法完全不变。

**优势**：
- 每个模块职责单一，便于测试和维护
- 新功能可独立模块开发，不影响其他模块
- 代码复用更清晰（其他脚本可单独 import 某模块）
- 调试时只需关注相关模块，无需阅读全部代码

## v1.3.1 — 2026-03-31

### 🐛 修复

#### 第五轮：贴士指数 JS 动态渲染 + 排位表正则匹配修复
- **问题1**：HKJC 贴士指数页面是 JS 动态渲染，静态 HTML 抓取无法获取完整数据
- **问题2**：排位表页面的 TR class 正则 `class="f_tac f_fs13"` 匹配失败，class 顺序可能是 `f_fs13 f_tac`
- **根因**：
  1. `fetch_tips_index()` 使用 `fetch_url()` 静态抓取，无法获取 JS 渲染后的表格数据
  2. `parse_race_entries()` 正则要求 class 必须为 `f_tac f_fs13` 精确顺序，但实际 HTML 中顺序不固定
- **修复**：
  1. 新增 `fetch_tips_index()` 使用 Playwright 动态加载贴士指数页面，等待 3 秒 JS 执行
  2. 新增 `fetch_url_with_playwright()` 函数用于获取需要 JS 渲染的页面
  3. `main()` 中排位表抓取改用 `fetch_url_with_playwright()`
  4. `parse_race_entries()` 正则改为 `[^\"]*f_tac[^\"]*f_fs13[^\"]*`（同时包含两个 class 即可）
  5. 新增后备马过滤：TD 数 < 12 的行被排除
  6. TD 列索引调整：[0]=马号, [2]=马名, [4]=负磅, [5]=骑师, [6]=档位, [7]=练马师, [10]=评分
  7. 新增 `_fetch_tips_index_static()` 备用解析（Playwright 不可用时）
- **验证**：2026/04/01 沙田第7场成功获取 12 匹马数据 + 12 条贴士指数（#3 銳一=3.0极热门等）
- **贴士指数评分生效**：狼来了 (#3) 贴士3.0 → 评分88；金玉良言(#6) 贴士6.4 → 评分65

#### 第四轮：排序键字段名错误（`final_score` vs `total_score`）
- **问题**：每场预测结果都固定为 `[1, 2, 3]`，无论哪场都一样
- **根因**：`daily_scheduler.py` 中排序键使用 `h.get("final_score", 0)`，
  但 `analyze_race.py` 实际输出的字段名是 `total_score`。
  所有马的 `final_score` 都返回默认值 0 → 得分相同 →
  Python 稳定排序保持原始列表顺序 → 马号 1, 2, 3 永远排在前面
- **修复**：`daily_scheduler.py` 第 266、273 行：`final_score` → `total_score`
- **验证**：修复后各场马号完全多样化，评分差异明显（37–62 分）

#### 第三轮：后备马混入 + 马名提取失败
- **问题**：预测结果混杂后备马（如傳承效應、飛雲），且马号与马名无法对应
- **根因**：
  1. TR 正则 `class="[^"]*f_fs\d+[^"]*"` 同时匹配正选马（`f_tac f_fs13`）和后备马（`bg_white f_tac f_fs13`）
  2. 马名提取正则 `href="...horse\?horseid=[^"]+[^>]*>` 中，`[^"]+` 在 href 闭合 `"` 后遇到紧跟的 `onclick=`（中间无空格），贪心匹配导致失败
- **修复**：
  1. TR 匹配改为精确 `class="f_tac f_fs13"`（排除 bg_white 后备马）
  2. 所有链接提取改用 `[^>]*>` 绕过 onclick 干扰：`r'horseid=([A-Za-z0-9_]+)[^>]*>([^<]+)</a>'`
- **验证**：各场正选马完全独立（无后备），马号与马名一一对应
